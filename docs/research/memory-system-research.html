★The short answer

YESdirection is right

Your instinct is not just possible — it's where the entire field has independently converged. The two most cited agent-memory systems of the last two years (Stanford's Generative Agents and HippoRAG) describe almost exactly what you sketched: importance/emotion weighting, associative graphs, and spreading-activation retrieval modeled on the hippocampus. You arrived at it from first principles. That's a strong signal.

What's validated: short-vs-long-term split, emotional/importance weighting, indexing memories by concept, a graph of linked experiences, neuron-like associative recall, learning a lesson once and recalling it forever — and your newest idea: short-term memory as a cache that "dreams" to garbage-collect itself while idle, promoting only what's worth keeping. A 2026 paper (SCM) implements exactly this. All of it has direct research backing and shipping implementations.

What needs correcting (4 things): ① Dijkstra is the wrong algorithm — recall is not a shortest-path problem; use spreading activation / Personalized PageRank. ② Pure graph isn't enough — the winning systems are hybrid (vectors + graph + time + symbolic index), because "dentist" must match "tooth pain" even with no explicit edge. ③ Don't node-ify every footstep — over-granular graphs become hairballs; store atomic-but-meaningful events. ④ Don't use pure LRU for eviction — it drops important-but-old lessons; weight eviction by importance + frequency too, the way modern caches (ARC, W-TinyLFU) do (§10).

Is it better than today's daily-log audit? Decisively, for the one thing you care about — recall. The current system is write-optimized and read-hostile. This is read-optimized.

1Auditing the memory system Friday has today

Before improving it, let's be precise about what exists and exactly where it hurts. Friday's memory lives at ~/.claude/.../memory/ and has four parts:

📌 MEMORY.md — the hand-written index

A flat bullet list, one line per fact, loaded into context every session. Today it has 3 entries. It is curated by hand, has no ranking, and every line is paid for in tokens on every turn whether relevant or not.

🗂️ feedback_*.md / friday-therapy/ — semantic memory

One fact per file with frontmatter (name, description, type) plus Why / How to apply. This is genuinely good — atomic, explained, the closest thing to true long-term lessons.

📅 daily/*.md — the episodic log

Time-stamped journal entries ("13:33 — Thread … Venkatesh asked for dashboard setup …"). 7 files, 35 lines total. This is the "log what happens every day" layer — and it's exactly where your recall problem lives.

🔗 Linking via [[name]]

The format already supports wiki-style links between memories. This is the seed of your graph — it just isn't traversed, ranked, or used at retrieval time. Right now it's decoration.

Where it breaks — the recall gap

The system is append-optimized and recall-hostile. Writing is trivial; finding is not. Concretely:

  • Retrieval is "load everything in MEMORY.md" or "grep the dailies." There is no relevance ranking. A query like "have we set up a member + admin dashboard for someone before, and what went wrong?" requires reading every daily file. At 35 lines that's fine. At 3,500 it's impossible — and the daily log only ever grows.
  • The dailies are a write-only graveyard. They're never read back because there's no index over them. Your dentist analogy is exact: you wrote down the visit, but you can't recall it without knowing the date — and you never know the date.
  • No salience. A throwaway "asked him to confirm phone number" sits at the same weight as a hard-won lesson that cost an hour of rework. Nothing decays, nothing gets promoted.
  • No associative jump. "Venkatesh" → "dashboard setup" → "the last admin-access bug" is a chain a human makes instantly. The current store has no edges to walk.
  • The index is manual and bounded. MEMORY.md is maintained by Claude writing a bullet by hand. It doesn't scale and it competes for context budget on every turn.
The reframe

"Memory audit" today means a human/agent re-reading the logs. What you actually want is a system that surfaces the right past memory at the right moment, unprompted — the way the smell of a drill comes back the instant you sit in the chair. That is a retrieval problem, not a logging problem. The fix is to keep writing what you write, and add a retrieval brain on top.

3The cognitive science your intuition is built on

Four findings from memory neuroscience justify the design. Each one becomes a concrete mechanism later.

① Emotion and surprise act as a "save this" flag

The amygdala, firing during emotionally or surprisingly intense events, biases the hippocampus to encode them more strongly — which is why "unexpected, isolated events, particularly with high emotionality, are better recalled."7 Your drill memory is an amygdala flag. Engineering translation: compute an importance/surprise score at write time and use it as a retention and ranking weight. (For an engineering agent, frame "emotion" as consequence/cost/surprise: a deploy that broke prod is "emotional" in exactly this sense.)

② Forgetting is a power-law decay — and it's a feature

Ebbinghaus showed memory strength decays as a power function of time; ACT-R formalizes this as a per-trace decay so that recency (recent = strong) and frequency (often-used = strong) jointly set an item's activation.5 Forgetting isn't a bug — it's how the brain keeps the relevant set small. Translation: decay scores over time; let unused, low-importance memories fade so retrieval stays sharp and cheap.

③ Two systems: fast episodic store, slow semantic store

Complementary Learning Systems theory: the hippocampus rapidly captures specific experiences; during offline periods (sleep) those traces are "replayed" and consolidated into neocortex as generalized knowledge, with replay probability biased by recency and frequency.6 Translation: a fast short-term buffer + a periodic consolidation job that promotes and generalizes the salient stuff into long-term semantic memory. This is your short-term/long-term split.

④ Retrieval is spreading activation, not search

In an associative network, a cue (e.g. "dentist") injects activation that radiates outward and decays with distance, with stronger links carrying more activation; the cascade lights up related items "multiple associations away" — feelings, lessons, warnings.8 This is precisely your "synapses forming" picture, and it is fundamentally different from finding a shortest path. More on that in §5.

4What's already been built — and what to steal from each

Seven systems define the space. Each solved part of your problem. The right design borrows the best idea from each rather than adopting any one wholesale.

🏛️ Generative Agents (Stanford, 2023) — the scoring formula

Smallville's agents store every experience as a natural-language line in a memory stream, then retrieve with a weighted score combining recency (exponential decay), importance (an LLM rates each memory 1–10), and relevance (embedding similarity to the current situation). Periodic reflections synthesize raw memories into higher-level insights that feed back in.2

Steal: the three-factor retrieval score and the reflection step. This is your importance/emotion weighting, made into one line of math.

🧠 HippoRAG (OSU, NeurIPS 2024) — your graph + pathfinding, validated

Explicitly built on the hippocampal indexing theory. The LLM is the "neocortex" (understanding), a knowledge graph is the associative index, and Personalized PageRank plays the role of the hippocampus — injecting activation at the query's entity nodes and letting it spread across the graph to rank what's relevant. Result: +20% on multi-hop QA, and 10–20× cheaper / 6–13× faster than iterative retrieval.3

Steal: this is the literal implementation of "store as a graph and use pathfinding to recall." The correct "pathfinding" is Personalized PageRank = spreading activation. Use it.

⏳ Zep / Graphiti (2024–25) — time as a first-class citizen

A temporal knowledge graph: every node and edge carries valid_at / invalid_at timestamps, so when a fact changes the old one is marked invalid rather than overwritten. This lets it answer "what did we believe last Tuesday?" and cleanly supersede stale facts. On the LongMemEval benchmark Zep scored 63.8% vs Mem0's 49.0% — a 15-point gap driven by exactly this temporal modeling.1

Steal: bi-temporal edges and explicit supersedes/contradicts relationships. This is how you handle "his phone number was wrong, then corrected" without lying later.

🗃️ A-MEM (2025) — Zettelkasten self-organizing notes

Each new memory becomes an atomic "note" with LLM-generated context, keywords, and tags. The system finds relevant existing notes and links them, and crucially lets memory evolve: a new note can update the descriptions/tags of older notes — the network refines itself over time. Ablations show both linking and evolution are load-bearing.4

Steal: auto-generated indexes per note, automatic link formation (your "synapses"), and the evolution step — exactly your "the brain decides to connect this to that."

💾 MemGPT / Letta (2023→) — the memory hierarchy

Treats the context window like RAM and external stores like disk, with explicit "paging." Three tiers: core/working memory (in context), recall/episodic (searchable conversation history), and archival/semantic (vector + KG cold storage). The agent uses tool calls to move data between tiers.9

Steal: the explicit short-term ↔ long-term tiering and the idea that the agent decides what to promote/page. This is your short-term vs long-term split, operationalized.

🔀 Mem0 (2025) — the pragmatic hybrid baseline

Dual store: a vector DB for fuzzy semantic search plus a graph for entity relationships, with an LLM "extract → update" pipeline. Easy to self-host, broad integrations. Loses to Zep on temporal queries but is the simplest production-grade starting point and proves the vector+graph hybrid is the mainstream answer.1

Steal: the extract-on-write pipeline and the confirmation that you want both vectors and a graph, not one or the other.

Pattern across all seven

Nobody picks graph or vectors or a log. The winners are hybrids: an episodic log for ground truth, embeddings for fuzzy entry, a graph for associative multi-hop, time for ordering and supersession, and an LLM for extraction and reflection. Your design should be a hybrid too.

5Three course corrections before you build

Correction 1 — Dijkstra is the wrong tool

Recall is not a shortest-path problem. Dijkstra finds the single cheapest route between two known nodes. But recall has no fixed destination — you have a cue ("dentist") and you want everything it lights up, ranked by how strongly. That's a one-to-many, decay-with-distance, strength-weighted cascade.

The correct algorithm is spreading activation (and its graph-ranking cousin Personalized PageRank, which is what HippoRAG uses). Inject "energy" at the cue node(s); it flows along edges proportional to edge strength, decaying each hop; the final activation level of every node is its recall score. Unlike Dijkstra: it's multi-source, it rewards many weak paths converging on a node, and it naturally produces a ranked set rather than one path.8

Correction 2 — pure graph loses fuzzy matching

A graph only connects what you explicitly linked. But "tooth pain," "root canal," and "that drilling sound" should all reach "dentist" even if you never drew those edges. Symbolic/graph indexes give precision and explainability; vector embeddings give fuzzy semantic reach. You need both: embeddings to find the entry nodes, the graph to expand from them. This is the hybrid every top system landed on — and it's why "vectorize it" and "use a graph" are not competing answers; they're two stages of one pipeline.

Correction 3 — don't node-ify every footstep

Your example listed "walking to the dentist / sitting in clinic / hearing the drill / coming home" as four memories. As a mental model, perfect. As storage, that granularity makes a hairball: millions of trivial nodes, edges that mean nothing, retrieval drowned in noise. A-MEM's lesson: store atomic but meaningful units — one note per episode ("dentist visit, 2026-05-20, painful root canal, dreaded it, glad it's done"), with structured attributes inside. Let granular detail live in the note, not as separate nodes.

What to keep exactly as you imagined it

One-way vs two-way edges (causal links are directed: cavity → root canal; "related-to" links are undirected) — keep this, it's correct and matters. Indexes per memory — keep. Synapses strengthening with co-activation — keep, that's Hebbian reinforcement and it's a real, cheap mechanism. Short-term vs long-term — keep, with a consolidation job between them. Learn-a-lesson-once — keep, that's reflection.

7The recall scoring model — the heart of it

Every candidate memory gets one number. This single formula is where "emotional weight, importance, frequency, recency, and context" all live. It's the Generative-Agents score, extended with frequency and spreading activation.

Start with all weights = 1 and normalize each term to [0,1] (Generative Agents did exactly this). Then tune on real Friday queries. That explainability is something a pure vector search cannot give you.

8Data model & schema

Memories are nodes; relationships are typed, weighted, optionally directed, optionally time-bounded edges. Below is a SQLite-flavored schema (rationale for SQLite in §11) that overlays your existing markdown without replacing it.

-- Each markdown memory becomes a row. The .md file stays the source of truth.
CREATE TABLE memory (
  id            TEXT PRIMARY KEY,        -- slug, matches the .md filename
  tier          TEXT,   -- 'episodic' | 'semantic' | 'procedural'
  body          TEXT,   -- the human-readable memory text
  source_path   TEXT,   -- e.g. daily/2026-05-18.md or feedback_*.md
  importance    REAL,   -- 0..1, LLM-rated consequence/surprise at write time
  created_at    INTEGER,
  last_used_at  INTEGER, -- drives recency
  use_count     INTEGER DEFAULT 0,    -- drives frequency
  valid_at      INTEGER, -- bi-temporal (Zep-style): when the fact became true
  invalid_at    INTEGER  -- NULL = still true; set when superseded
);

-- Entities mentioned across memories = the index ("glossary").
CREATE TABLE entity ( id TEXT PRIMARY KEY, name TEXT, kind TEXT );  -- person/repo/feature/concept
CREATE TABLE mention ( memory_id TEXT, entity_id TEXT ); -- inverted index: entity → memories

-- The graph. Directed when the relation is asymmetric; weight = synapse strength.
CREATE TABLE edge (
  src       TEXT,
  dst       TEXT,
  type      TEXT,   -- 'temporal_next' | 'caused' | 'about' | 'similar' | 'supersedes' | 'lesson_from'
  directed  INTEGER,-- 1 = one-way (causal), 0 = two-way (associative)
  weight    REAL,   -- strengthens via Hebbian reinforcement on co-activation
  PRIMARY KEY (src, dst, type)
);

-- Fuzzy entry points. sqlite-vec stores the vectors; FTS5 gives keyword search.
CREATE VIRTUAL TABLE mem_vec USING vec0(memory_id TEXT, embedding FLOAT[1024]);
CREATE VIRTUAL TABLE mem_fts USING fts5(memory_id, body);

Edge types — your "one-way and two-way links," typed

Directed (one-way) — asymmetric meaning

  • caused / led_to — cavity → root canal
  • supersedes — corrected phone # → old phone #
  • lesson_from — the rule → the episode that taught it
  • temporal_next — event A precedes event B

Undirected (two-way) — symmetric association

  • similar — semantically close memories
  • about (shared entity) — both mention "Venkatesh"
  • co_occurred — happened in the same thread/session
  • same_topic — both about "admin dashboard"

9Retrieval flow — how a query becomes a memory

Five stages. Stages 1–2 are the "fuzzy entry" (vectors + index); stages 3–4 are the "associative spread" (graph); stage 5 is the budget gate.

  1. Seed by similarity + index. Embed the incoming situation and run vector search (sqlite-vec) for the top-N similar memories; in parallel, extract entities ("Venkatesh," "admin dashboard") and look them up in the inverted index. Union → seed set. This solves the fuzzy "dentist ≈ tooth pain" problem.
  2. Inject activation at seeds. Each seed gets initial activation proportional to its similarity × importance. These are the cue nodes.
  3. Spread. Run spreading activation / Personalized PageRank for a few hops: activation flows along edges ∝ edge weight, decaying per hop. Convergent paths (a node reached from several seeds) accumulate — that's the "many weak signals add up" effect Dijkstra can't give you.8
  4. Score & rank. Combine the §7 formula: w₁·recency + w₂·importance + w₃·relevance + w₄·activation + w₅·frequency. Apply the temporal filter — drop memories whose invalid_at is set (superseded facts) unless the query is explicitly historical.
  5. Budget gate. Return only the top-k (e.g. 5–8) that fit a token budget. Over-retrieval poisons context as badly as no memory — keep it tight. Bump last_used_at/use_count and Hebbian-strengthen the edges among co-retrieved memories.
Worked example (real Friday data)

Situation: "Venkatesh wants an admin dashboard set up." → Seeds: the 2026-05-18 episodic note (entity "Venkatesh," topic "admin dashboard") + the semantic lesson verify-fridays-claims (similar topic: setup outcomes). → Spread reaches the dm-on-blocking-decisions feedback via a "blocked-on-judgment-call" link (the phone number was short a digit — a blocking call). → Friday recalls, unprompted: "last time this stalled on an unverified phone number; confirm the full number and DM Anmol if it blocks." That's the dentist-chair moment.

10Dreaming — short-term memory as a cache that gets garbage-collected

Your LRU-cache instinct is the missing piece that makes the short/long-term split actually work, and it's well-founded. The day's memories live in a bounded short-term cache; while Friday is idle it "dreams" — a sleep cycle that replays the cache, promotes what's worth keeping into long-term, and garbage-collects the rest. A 2026 paper, Sleep-Consolidated Memory (SCM), implements precisely this: a limited-capacity working memory, importance tagging, offline NREM/REM sleep stages, and intentional value-based forgetting.10

Short-term memory is a bounded cache

The day's episodic memories shouldn't grow forever — they sit in a fixed-capacity buffer (capped by item count or token budget). When it fills, something must be evicted. That is a cache-eviction problem, and your LRU framing is the right starting instinct: recency-based forgetting is real and biologically grounded (Ebbinghaus's curve5). But the policy needs one upgrade.

Correction 4 — don't use pure LRU

Pure LRU evicts strictly by recency, so a hard-won lesson learned at 9am gets thrown out the moment newer trivial chatter arrives — exactly the failure you're trying to avoid (the drill you can't forget would be evicted by the stranger you just passed). Production caches abandoned pure LRU long ago: LFU adds frequency, ARC adapts between recency and frequency, and W-TinyLFU (the policy behind Caffeine, the state-of-the-art JVM cache) combines both for the highest hit rates.11 Use a salience-weighted eviction: when the cache is full, evict the item with the lowest §7 value score (recency + importance + frequency). That's ARC/W-TinyLFU thinking — and it happens to be exactly what ACT-R activation already computes. The cache keeps what's recent, frequent, or important.

"Dreaming" — the sleep cycle that GCs the cache

Garbage collection doesn't run on every write (too expensive — it's LLM work). It runs while Friday is idle, like sleep. SCM splits this into stages that map cleanly onto our pipeline:10

🌙 NREM — replay & promote

Re-score the day's cached episodes by the §7 value. High importance × frequency get "replayed" and promoted toward semantic memory (CLS consolidation6). A one-off "asked for phone number" fades; a pattern seen 3× becomes a rule.

💭 REM — reflect & extract lessons

Ask the LLM: "what general lessons do today's events imply?" Write them as feedback_*.md with a lesson_from edge back to the episodes. This is the hot-pan mechanism — and the core idea of Reflexion: verbally reflect on failure, store the reflection, do better next time.215

🗑️ Forget — evict & archive

Items below the retention bar are evicted from the active cache (salience-weighted, not pure LRU) and moved to cold archive — never hard-deleted. Forgetting keeps the hot graph small, fast, and relevant: a feature, per Ebbinghaus/ACT-R and SCM. MemoryBank implements exactly this — Ebbinghaus decay on memory strength, reinforced on each recall.51014

🔗 Strengthen — evolve links

Re-run seed-and-link over the day's additions (A-MEM evolution4): form new edges, strengthen co-activated ones (Hebbian: "neurons that fire together wire together"), and mark contradictions with supersedes + a fresh invalid_at.

The retention rule — how an item earns long-term storage

This answers your "based on the research, how do we retain them for long term." During a dream, a short-term item is promoted to long-term (semantic) memory if any of these hold:

  • Value ≥ promotion threshold — high importance × recency × frequency clears the bar.
  • Reinforced ≥ k times that day — it kept coming up, so it kept mattering (frequency / Hebbian reinforcement).
  • High importance / surprise — the amygdala flag: a deploy broke prod, a user was upset, a hard correction landed.7
  • Generalized into a rule — a reflection turned it into a reusable lesson (a lesson_from edge was created).

Everything else is evicted from the active cache but archived, not deleted — the markdown and git history remain. "Garbage collection" here means "remove from the hot working set," like paging cold data to disk. If an archived memory later proves relevant, a vector/index hit can re-admit it — exactly a cache miss that pulls from disk and warms back up.

Why dream nightly, not on every write

Reflection, re-linking, and re-scoring are LLM calls — too costly to run per Slack turn. Batching them into an idle-time "dream" mirrors biological sleep consolidation and keeps per-turn latency and spend low. Daytime writes stay cheap (extract entities + embed + append to the cache); the expensive "thinking about it" — and the garbage collection — happens offline. Friday already has the cron/hook machinery to schedule this.

12MVP roadmap — ship value at every phase

Each phase is independently useful and non-destructive. You could stop after Phase 2 and already have solved the recall problem.

Phase 0 · ~½ day · foundation

Structure what you already write

Add minimal frontmatter to episodic notes too (entities, importance 1–10, type). Keep writing markdown by hand exactly as today. No retrieval changes yet — just make the data machine-readable. Win: nothing breaks; future phases have clean inputs.

Phase 1 · ~2 days · the index

SQLite overlay: FTS + embeddings, rebuilt from files

A script walks memory/**/*.md, populates memory, mem_fts, mem_vec. A recall(query, k) function does vector + keyword search and returns top-k. Wire it into Friday's prompt assembly. Win: the daily-log graveyard becomes searchable — the core recall problem is solved at a basic level here.

Phase 2 · ~3 days · the graph + scoring

Entities, edges, and the 5-term score

Extract entities on write; build entity/mention/edge tables. Implement the §7 scoring formula and §9 spreading activation (a bounded BFS with decay, or NetworkX PageRank in-process). Win: associative recall — "Venkatesh" surfaces the related blocking-decision lesson. This is the dentist-chair behavior.

Phase 3 · ~2 days · dreaming & cache GC

Bounded short-term cache + nightly "dream"

Cap the episodic buffer; evict by lowest §7 score (salience-weighted, not pure LRU). Cron "dream" job: replay & re-score the day, extract lessons into feedback_*.md, apply the retention rule, promote the salient few to long-term, archive the rest. Win: the system learns and forgets on its own — short→long-term promotion is automatic instead of hand-curated, and memory stays bounded.

Phase 4 · ongoing · refinement

Hebbian reinforcement, eval, tuning

Strengthen co-retrieved edges; build a small recall@k eval set from real threads (LongMemEval-style1); tune the five weights; add a "why retrieved" trace for the audit view. Win: measurable, improving, explainable recall.

Scope discipline (per your own CLAUDE.md)

Resist building Phase 2–4 first. Phase 1 alone — searchable embeddings over the markdown you already write — captures most of the value for a fraction of the effort. The graph and consolidation are multipliers, not prerequisites. Build the index, prove recall improves on real queries, then add the brain.

13Risks, edge cases & how to handle them

RiskWhy it bitesMitigation
Contradiction / stale facts"Phone is 9 digits" then "phone corrected." Naive recall surfaces the wrong one.Bi-temporal edges (Zep): supersedes + invalid_at. Filter invalid facts at retrieval unless query is historical.1
Context poisoning / false memoryA wrong memory, once stored, gets recalled and reinforced — a confidently-held mistake.Provenance on every memory (source_path); confidence scores; let the consolidation step flag low-confidence or unverified items (ties to your "verify Friday's claims" rule).
Over-retrievalStuffing 30 memories into context degrades the answer as much as zero.Hard top-k + token budget gate (§9.5). Fewer, higher-scored memories beat many.
Graph hairballToo many nodes/edges → slow traversal, noisy activation.Atomic-but-meaningful nodes (Correction 3); cap edges per node; decay/prune in consolidation.
Premature forgettingDreaming evicts something that turns out important later.Archive, never hard-delete (re-admit on a later hit); keep the promotion threshold conservative; high-importance items are exempt from eviction (Correction 4 / §10).
LLM extraction costEntity/importance/reflection are model calls; per-turn they add latency & spend.Cheap path on write (embed + regex/NER for entities); expensive reflection batched nightly.
Embedding driftChanging the embedding model invalidates stored vectors.Store model id with each vector; rebuild index from markdown (it's derived, not authoritative) when the model changes.
Importance is subjectiveThe LLM's 1–10 rating is noisy and can drift.Calibrate against outcomes (did this memory ever get usefully recalled?); let frequency-of-use correct mis-rated importance over time.
Privacy / leakageMemories carry PII (phone numbers, emails) and personal context.Keep the store local to Friday's host; never ship raw memory to external services; redact PII in any shareable export.

14Is it actually better? Measuring recall & the audit angle

"Whether this is more efficient than the current memory audit" deserves a real answer, not a vibe. Here's how to prove it — and what "audit" should mean going forward.

Efficiency vs today

  • Recall: today = O(read every daily file) and degrades to "impossible" as logs grow. Proposed = O(vector top-N + bounded graph spread), constant-ish regardless of total size.
  • Token cost per turn: today loads all of MEMORY.md unconditionally; proposed loads only the top-k relevant — cheaper as memory grows, not more expensive.
  • The trade: you pay a one-time indexing cost on write + a nightly consolidation cost. For recall quality that's a clear win past a few dozen memories.

How to measure it

  • Build a recall@k eval: 20–30 real questions from past threads with the memory that should surface. Measure how often it's in the top-k. Re-run as you tune weights.
  • Borrow query types from LongMemEval1: single-hop, multi-hop, temporal, knowledge-update. Friday's hardest cases are temporal + update (the phone-number correction).
  • The closest public benchmark to Friday is LoCoMo16 — 300-turn, up-to-35-session dialogues built on persona + event graphs, scored on QA, summarization, and temporal/multi-hop reasoning. Build a small LoCoMo-style set from real Slack threads to track recall over releases.
  • Track "useful recall rate": of memories surfaced, how many did Friday actually use? Low → over-retrieving; tune the budget.
Redefining "memory audit"

Today an audit means a human re-reads logs to reconstruct what happened. With this system, audit becomes self-service and explainable: every retrieval carries a why trace (which seeds, which edges, which score terms dominated) and every memory carries provenance (source file, created/valid timestamps, confidence). You can ask "what does Friday know about Venkatesh and why does it think the phone number matters?" and get a sourced, ranked, timestamped answer instead of a grep. That's the real upgrade — not just faster recall, but auditable recall.

📚Further reading — the papers to actually read, in order

You asked what we'd missed and what to read. Here's both: five concepts the first draft under-covered (now folded in), then a curated, ordered path through the literature. Read top-down — each tier assumes the one above.

Five things the first draft under-covered (now added)

① Hierarchical summarization (RAPTOR, GraphRAG) — multi-level summaries so Friday can answer "what are the recurring themes across all my threads?", not just point lookups. The original "reflection" was single-level. ② Surprise as a computable importance signal (EM-LLM, Titans) — you don't have to ask the LLM to rate 1–10; surprise = how unpredictable the event was (negative log-likelihood), which you can measure. ③ The forgetting curve has a direct LLM precedent (MemoryBank) — §10's decay isn't speculative. ④ Procedural memory = a skill library of executable code (Voyager), not just prose notes. ⑤ Use a real benchmark (LoCoMo) — its persona+event-graph dialogues are the closest public analog to Friday's Slack threads.
