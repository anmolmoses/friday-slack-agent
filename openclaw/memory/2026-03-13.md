### PR Review Learnings (from #tech-pr-reviews thread)
- Anmol told me to learn from Junior's PR review of gx-backend #2848 (Event assistant for event creation)
- **What Junior did well:**
  - Gave a structured review: outstanding items, new findings, what looks good
  - Referenced exact files and line numbers (`models/event/event.ts` line 394)
  - Provided specific code fixes (not vague "fix this" — actual replacement code)
  - Linked directly to the PR comment (`#r2030069498`)
  - Verified the fix before approving — didn't just take the dev's word
  - tl;dr summary at the end for quick scanning
- **Standards to follow for PR reviews:**
  1. Structure: outstanding → new findings → looks good → tl;dr
  2. Always reference exact files, lines, and provide code snippets
  3. Link to specific PR comments when pointing out issues
  4. Verify fixes are actually pushed before approving
  5. Be clear about blockers vs housekeeping items
- This is the bar for when I route PRs to Gilfoyle or handle reviews myself
